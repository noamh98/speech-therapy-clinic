                         INITIAL SECURITY ISSUES FOUND
================================================================================

🔴 CRITICAL ISSUES (8):
─────────────────────────────────────────────────────────────────────────────

1. GEMINI API KEY EXPOSED IN FRONTEND
   Location: src/pages/AIAssistant.jsx:111
   Issue: const apiKey = import.meta.env.VITE_GEMINI_API_KEY;
   Risk: API key visible in DevTools → Network tab
   Severity: CRITICAL - Anyone could steal the key and use it on their account
   Fix: Move API calls to backend Cloud Function

2. EMAIL-BASED ADMIN CHECK
   Location: src/context/AuthContext.jsx:73
   Issue: const isAdmin = profile?.role === 'admin' || user?.email === ADMIN_EMAIL;
   Risk: Email spoofing could grant admin access to attackers
   Severity: CRITICAL - Email can be changed, bypassing security
   Fix: Remove email check, use only profile.role field

3. ADMIN ROLE SET FROM FRONTEND
   Location: src/pages/AdminUsers.jsx:30-42
   Issue: User document created directly with role field from browser
   Risk: Attacker could set themselves as admin via DevTools
   Severity: CRITICAL - No backend validation of role assignment
   Fix: Disable frontend user creation, use Cloud Function only

4. EMAIL INSTEAD OF UID IN QUERIES
   Location: src/services/templates.js:11
   Issue: where('therapist_email', '==', user?.email || '')
   Risk: Attacker could hijack another user's data by changing email in database
   Severity: CRITICAL - Email is mutable, UID is not
   Fix: Change to where('ownerId', '==', user.uid)

5. XSS VULNERABILITY IN MARKDOWN
   Location: src/pages/AIAssistant.jsx:379
   Issue: <ReactMarkdown>{m.content}</ReactMarkdown>
   Risk: Prompt injection could inject malicious JavaScript
   Severity: CRITICAL - AI responses are rendered without sanitization
   Fix: Add rehype-sanitize plugin to filter HTML

6. FIRESTORE SECURITY RULES - EMAIL BASED
   Location: firestore.rules:12, 30, 37
   Issue: function isAdmin() { return request.auth.token.email == 'noamh98@gmail.com'; }
   Risk: Hardcoded email, rules check email instead of database role
   Severity: CRITICAL - Email in auth token could be forged
   Fix: Check database for role field instead

7. FIRESTORE RULES - MISSING UID CHECK
   Location: firestore.rules - all collections
   Issue: Rules use therapist_email instead of ownerId
   Risk: No proper ownership validation of documents
   Severity: CRITICAL - Data isolation broken
   Fix: Add ownerId field to all collections, check UID in rules

8. FIREBASE STORAGE - PUBLIC RECEIPTS
   Location: storage.rules - not defined properly
   Issue: Payment receipts accessible without proper UID check
   Risk: Sensitive financial documents exposed
   Severity: CRITICAL - Confidential data leak
   Fix: Add per-user storage rules with UID validation

🟠 HIGH PRIORITY ISSUES (4):
─────────────────────────────────────────────────────────────────────────────

9. RATE LIMITING MISSING
   Issue: No throttling on Gemini API calls
   Risk: User could spam thousands of messages, causing DoS and high costs
   Fix: Add 2-second throttle between messages

10. ADMIN INVITATION - NO BACKEND VALIDATION
    Issue: Role assignment in AdminUsers.jsx not validated server-side
    Risk: Direct database writes without Cloud Function
    Fix: Create Cloud Function to validate and assign roles

11. INTAKEOFRMS & OTHER SERVICES - EMAIL QUERIES
    Issue: Multiple services use email instead of UID for querying
    Risk: Data ownership not properly validated
    Fix: Audit and update all services to use UID

12. STORAGE RULES - INCOMPLETE
    Issue: storage.rules exists but not fully configured for multi-user
    Risk: Potential cross-user data access
    Fix: Add proper ownerId-based access controls

STEP 5: UPDATED FIRESTORE SECURITY RULES
─────────────────────────────────────────────────────────────────────────────
File: firestore.rules

Critical Changes:

A) ADMIN CHECK - Changed from email to role field
   Before: function isAdmin() {
             return isAuth() && request.auth.token.email == 'noamh98@gmail.com';
           }

   After:  function isAdmin() {
             return isAuth() && get(/databases/$(database)/documents/
                                     users/$(request.auth.uid)).data.role == 'admin';
           }

   Why: Email in auth token can be forged. Role in database is authoritative.

B) OWNERSHIP CHECK - Changed to use UID
   Before: resource.data.therapist_email == request.auth.token.email
   After:  resource.data.ownerId == request.auth.uid

   Why: UID is immutable, email is mutable. Prevents hijacking.

C) APPLIED TO ALL COLLECTIONS
   Updated: patients, appointments, treatments, progress, intakeForms,
            templates, notifications, payments

   Each now validates: ownerId field matches current user's UID

D) ADDED UPDATE PROTECTION
   Added check: request.resource.data.ownerId == resource.data.ownerId

   Why: Prevents users from changing the owner field of documents

E) STORAGE RULES ADDED
   New rules for receipts with path-based UID isolation:
   match /receipts/{ownerId}/{allPaths=**} {
     allow read, write: if request.auth.uid == ownerId;
   }

STEP 6: ADDED RATE LIMITING
─────────────────────────────────────────────────────────────────────────────
File: src/pages/AIAssistant.jsx:160, 189

Added state: const [lastMessageTime, setLastMessageTime] = useState(0);

Added check in send() function:
  const now = Date.now();
  if (now - lastMessageTime < 2000) {
    setError('⏳ אנא חכה 2 שניות בין הודעות');
    return;

  setLastMessageTime(now);

Prevents: Users from spamming the API with thousands of requests

STEP 7: BUILT CLOUD FUNCTION - CALLGEMINI
─────────────────────────────────────────────────────────────────────────────
File: functions/index.js - exports.callGemini

Purpose: Proxy requests to Gemini API, keeping API key secure on backend

Implementation Details:

1. Authentication Check:
   ✅ if (!context.auth) throw HttpsError('unauthenticated')

   Ensures only logged-in users can call this function

2. Input Validation:
   ✅ Validate messages array, systemPrompt string
   ✅ Limit message history to 50 messages (prevent abuse)
   ✅ Check every message has role and content fields

   Prevents: Malformed requests, memory exhaustion, injection attacks

3. Secret Manager Integration:
   ✅ Configured with: functions.runWith({ secrets: ['GEMINI_API_KEY'] })
   ✅ Accesses via: const apiKey = process.env.GEMINI_API_KEY;

   Ensures: API key never exposed in frontend code

4. Gemini API Logic:
   ✅ Converts messages to Gemini format
   ✅ Maintains conversation history
   ✅ Sends last message to continue chat
   ✅ Returns response to frontend

   Benefit: Frontend never touches API key, all calls authenticated

5. Error Handling:
   ✅ Catches and logs errors
   ✅ Returns generic error to prevent info leakage
   ✅ Throws HttpsError which is automatically handled by Firebase

STEP 8: BUILT CLOUD FUNCTION - INVITEUSER
─────────────────────────────────────────────────────────────────────────────
File: functions/index.js - exports.inviteUser

Purpose: Safely invite new users with role assignment via Admin SDK

Security Features:

1. Triple Authentication Check:
   ✅ Verify user is logged in (context.auth)
   ✅ Verify user exists in Firestore (getDoc)
   ✅ Verify user has admin role (role === 'admin')

   Three checks prevent any unauthorized invitation

2. Email Validation:
   ✅ Regex pattern: /^[^\s@]+@[^\s@]+\.[^\s@]+$/
   ✅ Prevents: Invalid emails, injection attempts

3. Role Assignment - ADMIN SDK ONLY:
   ✅ Uses admin.firestore() which bypasses Firestore rules
   ✅ Runs on server with full admin privileges
   ✅ Ensures: Roles can only be set by legitimate admin calls

   Why Admin SDK? Frontend calls would be subject to Firestore rules
   which prevent users from creating admin accounts.

4. Audit Trail:
   ✅ Records: invitedAt (timestamp), invitedBy (admin's UID)
   ✅ Useful: For tracking who invited whom and when

🔴 CRITICAL THREATS (Now Fixed):

6. Firestore Data Exposure
   Before: Rules used email, no UID checks
   After: All access validated by UID + role

7. Storage Leaks
   Before: Any authenticated user could access any file
   After: Per-user UID-based isolation

8. DoS via API Spamming
   Before: No rate limiting
   After: 2-second throttle between messages

🟠 HIGH SEVERITY THREATS (Now Fixed):

9. Hardcoded Credentials in Code
   Before: ADMIN_EMAIL and API keys in source
   After: Externalized to environment and Secret Manager

10. Missing Input Validation
    Before: No validation on Cloud Function inputs
    After: Comprehensive input checks for all params

🔵 MEDIUM RISK THREATS (Now Mitigated):

11. Authorization Bypass
    Before: Single-layer auth (email or role)
    After: Multiple validation checks (auth + role + UID)

    After: invitedAt + invitedBy recorded for all invitations
